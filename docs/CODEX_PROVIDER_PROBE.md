# Codex provider probe specification

Status: proposed provider probe, not a promoted SentAInce provider.

## Decision boundary

Codex is the preferred near-term OpenAI integration target for local SentAInce because it has host-local
hooks and host-local MCP configuration. This avoids the ChatGPT Secure MCP Tunnel blocker, where connector
visibility depends on Platform organization, ChatGPT workspace association, tunnel identity, runtime keys,
and workspace permissions.

This PR does not claim full Codex support. It defines the measured probe that must pass before Codex can be
listed beside the already-supported hook providers.

## Integration surfaces

Codex maps onto the existing SentAInce split:

| SentAInce surface | Codex surface | Probe claim |
|---|---|---|
| Somatic gate and audit | `PreToolUse` hook | A planted destructive command is denied before execution. |
| Consequence observation | `PostToolUse` hook | Tool output, status, and identifiers are sufficient for outcome classification. |
| Prompt classification and memory splice | `UserPromptSubmit` hook | Prompt text can seed goal-class state and inject earned context. |
| Session initialization | `SessionStart` hook | Session identity and startup context can initialize SentAInce state. |
| Sleep / consolidation | `PreCompact` and `PostCompact` hooks | Manual and auto compaction events are observed and can set resplice/consolidation state. |
| Read-only earned-memory recall | Codex MCP server config | Codex can call the existing read-only MCP recall tools without mutating memory. |

## Provider shape

The implementation should be additive:

- Add `codex` as an explicit provider in `exocortex/adapter.py`; do not silently treat Codex as Claude.
- Keep the internal hook contract Claude-shaped where possible, because Codex accepts
  `hookSpecificOutput.permissionDecision = "deny"` and `hookSpecificOutput.additionalContext`.
- Normalize Codex fields explicitly:
  - `SessionStart`: session/thread identifiers and startup metadata.
  - `UserPromptSubmit`: `prompt`, `turn_id`, and permission mode.
  - `PreToolUse`: `tool_name`, `tool_input`, `tool_use_id`, `turn_id`, and permission mode.
  - `PostToolUse`: `tool_name`, `tool_input`, `tool_response`, `tool_use_id`, and outcome-bearing output.
  - `PreCompact` / `PostCompact`: `trigger` values `manual` and `auto`.
  - MCP tools: canonical tool names such as `mcp__server__tool` and their argument payloads.
- Treat `PermissionRequest` as a separate event if SentAInce later participates in approval prompts. It
  should not be collapsed into normal `PreToolUse` gating without a separate policy decision.

## Hook configuration sketch

The first probe can use project-scoped Codex config after the normal trust review flow. Exact paths should
be generated by the installer or test harness rather than hand-written by users.

```toml
[[hooks.SessionStart]]
matcher = "startup|resume"
[[hooks.SessionStart.hooks]]
type = "command"
command = "python -m exocortex.hook SessionStart --provider codex"

[[hooks.UserPromptSubmit]]
[[hooks.UserPromptSubmit.hooks]]
type = "command"
command = "python -m exocortex.hook UserPromptSubmit --provider codex"

[[hooks.PreToolUse]]
matcher = "^(Bash|apply_patch|mcp__.*)$"
[[hooks.PreToolUse.hooks]]
type = "command"
command = "python -m exocortex.hook PreToolUse --provider codex"

[[hooks.PostToolUse]]
matcher = "^(Bash|apply_patch|mcp__.*)$"
[[hooks.PostToolUse.hooks]]
type = "command"
command = "python -m exocortex.hook PostToolUse --provider codex"

[[hooks.PreCompact]]
matcher = "^(manual|auto)$"
[[hooks.PreCompact.hooks]]
type = "command"
command = "python -m exocortex.hook PreCompact --provider codex"

[[hooks.PostCompact]]
matcher = "^(manual|auto)$"
[[hooks.PostCompact.hooks]]
type = "command"
command = "python -m exocortex.hook PostCompact --provider codex"
```

> **Intake note (verified against the tree):** `exocortex.hook` today dispatches `PreToolUse`, `PostToolUse`,
> `UserPromptSubmit`, `SessionStart`, and `PreCompact` — it has **no `PostCompact` branch yet**, so a wired
> `PostCompact` hook is a no-op (exit 0, no handler) until one is added. The `--provider` flag and
> `EXOCORTEX_PROVIDER` are already read (`exocortex/hook.py`, `exocortex/adapter.py::detect`), which today
> resolves `cursor`/`claude` and defaults everything else to `claude` — this is exactly why `codex` must be
> added explicitly. The first code PR should add the `PostCompact` handler (or scope it out with observed
> evidence) alongside the `codex` adapter path.

## MCP recall configuration sketch

The Codex read side should reuse `exocortex/mcp_server.py`. This is recall only: no tau deposits, no sigma
writes, no colony/wiki mutation, and no somatic veto.

```toml
[mcp_servers.sentaince-memory]
command = "python"
args = ["-m", "exocortex.mcp_server"]
env = { EXOCORTEX_STATE_DIR = "/absolute/path/to/project/.claude/exocortex" }
```

The probe should exercise at least:

- `memory_status(repo="")`
- `recall_for_prompt(prompt, repo="", cls="")`
- one abstaining recall path, proving no memory is invented
- before/after file hashes for the memory store, proving MCP retrieval does not mutate memory

## Evidence packet required before promotion

Create `results/codex_provider_probe_v1/FINDINGS.md` with raw artifacts or links to raw artifacts showing:

| Requirement | Acceptance bar |
|---|---|
| `PreToolUse` fires for `Bash` | Audit row includes `event=PreToolUse`, `tool=Bash`, command text, and action. |
| Lethal command denial | A planted command such as a harmlessly-targeted destructive pattern is denied before execution; the target remains unchanged. |
| `apply_patch` visibility | Hook audit records an `apply_patch` event, or the results explicitly scope it out with observed evidence. |
| MCP tool visibility | Hook audit records a configured MCP tool call by canonical Codex MCP tool name. |
| `PostToolUse` consequence shape | Captured payload includes enough response/status data to classify `ok` vs `fail`. |
| `UserPromptSubmit` injection | Additional context is accepted as model-visible developer context for the same turn. |
| `SessionStart` state seed | A fresh session creates SentAInce session state and an audit row. |
| `PreCompact` and `PostCompact` | Both events are observed for `manual` or `auto`; final promotion needs both trigger values if Codex exposes them in practice. |
| MCP recall read-only invariant | Recall tools return data and leave memory files byte-identical. |
| Provider isolation | Existing `claude` and `cursor` provider behavior remains unchanged. |

## Non-goals

- Do not promote ChatGPT Apps or Secure MCP Tunnel as the default SentAInce OpenAI path.
- Do not claim Codex is a complete enforcement boundary until the live denial and event-coverage evidence
  exists.
- Do not deposit memory from MCP retrieval, prompt injection, human tags, or tool frequency.
- Do not rely on `permissionDecision = "ask"` for safety behavior; use hard deny for vetoes.
- Do not claim `PostToolUse` can undo side effects. It can only observe or alter feedback after the tool has
  run.

## Review checklist

- The implementation is dormant unless `EXOCORTEX_PROVIDER=codex` or installer-generated Codex config is in
  use.
- The first code PR includes focused adapter tests outside the locked top-level `tests/` evidence suite.
- The probe run captures raw hook stdin/stdout samples with secrets redacted.
- The PR body labels Codex as `probe` or `experimental`, not `supported`.
- Documentation cross-links this file from MCP/provider docs only after the first probe result exists.
